# Архитектура приложения Bank_Rest

## Общие принципы
- **Монолитная архитектура** с четким разделением на модули по функциональности и уровням доступа (admin, private, auth).
- **Единая реляционная база данных** (PostgreSQL) для всех модулей, согласованность через ACID-транзакции.
- **Безопасность**: JWT-аутентификация, ролевая модель (ADMIN, USER), проверка прав доступа на уровне контроллеров (`@PreAuthorize`) и сервисов (валидация владельца ресурса).
- **Шифрование конфиденциальных данных**: номера карт хранятся в зашифрованном виде (`EncryptionUtil`), при выводе маскируются (`CardMaskUtil`).
- **Идемпотентность операций**: ключевые операции (создание карты, перевод) защищены от дублирования на уровне бизнес-логики и БД.

## Модули приложения

### 1. Модуль аутентификации (Auth)
- **Контроллер**: `AuthController`
- **Функции**: регистрация, вход, обновление токенов, проверка уникальности username.
- **Особенности**: при успешной аутентификации возвращает access и refresh JWT-токены, а также данные пользователя.

### 2. Модуль управления пользователями
#### 2.1 Admin – управление пользователями
- **Контроллер**: `AdminUserController`
- **Доступ**: только `ADMIN`
- **Функции**: создание, просмотр всех, поиск, обновление, блокировка/активация, удаление пользователей.
- **Сервис**: `UserAdminService` → делегирует `UserOperationService`

#### 2.2 Private – управление своим профилем
- **Контроллер**: `PrivateUserController`
- **Доступ**: авторизованные пользователи (любой с JWT)
- **Функции**: получение и обновление своего профиля.
- **Сервис**: `UserPrivateService` → использует `UserOperationService`

### 3. Модуль управления картами
#### 3.1 Admin – карты
- **Контроллер**: `AdminCardController`
- **Доступ**: `ADMIN`
- **Функции**: создание карты для любого пользователя, просмотр всех карт (с пагинацией), просмотр карт конкретного пользователя, блокировка/активация/удаление карты по ID.
- **Сервис**: `CardAdminService` → делегирует `CardOperationService`

#### 3.2 Private – свои карты
- **Контроллер**: `PrivateCardController`
- **Доступ**: авторизованные пользователи
- **Функции**: просмотр своих карт, получение баланса, блокировка своей карты, перевод между своими картами.
- **Сервис**: `CardPrivateService` → добавляет проверку владельца, затем делегирует `CardOperationService`

### 4. Модуль транзакций (Admin)
- **Контроллер**: `TransactionController`
- **Доступ**: `ADMIN`
- **Функции**: просмотр всех транзакций (с пагинацией), просмотр транзакции по ID.
- **Сервис**: `TransactionServiceImpl` (работа напрямую с репозиторием)

## Слой сервисов (бизнес-логика)

### `UserOperationService`
- Содержит основную логику работы с пользователями: создание, обновление, блокировка, поиск.
- Работает с `UserRepository`, `RoleRepository`, `PasswordEncoder`.
- Используется как `UserAdminService`, так и `UserPrivateService`.

### `CardOperationService`
- Ядро операций с картами: создание, блокировка, активация, удаление, перевод, получение по ID и по владельцу.
- Взаимодействует с `CardRepository`, `UserRepository`, `TransactionRepository`, `EncryptionUtil`, `CardMaskUtil`.
- Транзакции: перевод выполняется атомарно в рамках одной транзакции БД (обновление балансов двух карт + создание записи транзакции).
- Генерация номера карты (16 цифр) с проверкой уникальности после шифрования.

### `TransactionServiceImpl`
- Простой сервис для чтения транзакций администратором.

### Вспомогательные сервисы
- `CardPrivateService` и `CardAdminService` — обёртки над `CardOperationService`, добавляющие проверки доступа.
- `UserPrivateService` и `UserAdminService` — аналогично для пользователей.

## Слой доступа к данным (DAO)
- **Репозитории Spring Data JPA**:
    - `UserRepository`, `CardRepository`, `TransactionRepository`, `RoleRepository`
- **Сущности**:
    - `User` (id, username, password, isActive, roles)
    - `Card` (id, encryptedNumber, owner, status, expiryDate, balance)
    - `Transaction` (id, fromCard, toCard, amount, createdAt)
    - `Role` (id, name)
- Связи: `@ManyToMany` (User-Roles), `@ManyToOne` (Card-User, Transaction-Card)

## Безопасность
- **JWT**: токены генерируются и валидируются `JwtTokenProvider`.
- **Роли**: `ROLE_ADMIN`, `ROLE_USER`. Проверка через `@PreAuthorize` в контроллерах.
- **Проверка владельца**: в `CardPrivateService` методы `validateCardOwner` гарантируют, что пользователь работает только со своими картами. При нарушении выбрасывается `CardAccessException`.
- **Аутентификация**: стандартный `AuthenticationManager` с `UserDetailsService`, загружающим пользователя из БД.

## Финансовые операции
- **Перевод между картами**:
    - Проверка: карты существуют, принадлежат одному пользователю (в Private), достаточно средств, карты разные.
    - Атомарность: обновление баланса `fromCard` (subtract) и `toCard` (add) в одной транзакции, затем создание `Transaction`.
    - При ошибке (недостаточно средств) – откат транзакции, исключение `TransferException`.
- **Блокировка/активация** – просто меняет статус карты.
- **Создание карты**:
    - Генерация номера, шифрование, проверка уникальности зашифрованного номера.
    - Привязка к существующему пользователю.

## Заключение
Приложение построено как классический монолит со строгим разделением ответственности. Оно обеспечивает безопасное управление картами и пользователями, финансовые операции выполняются атомарно, конфиденциальные данные защищены шифрованием. Архитектура позволяет легко расширять функциональность и поддерживать код.
